코드 컨벤션

Posted by     "진보라" on Monday, May 9, 2022

코드 컨벤션은 읽기 용이하고 관리하기 쉽게 규정한 일관된 코드 스타일이다. 코드 컨벤션이 코드의 성능을 끌어올리지 않고 깐깐한 규칙을 하나하나 따르다보면 오히려 시간을 더 잡아먹을 수도 있다. 그럼에도 왜 협업에서 코드 컨벤션을 준수하라고 할까?

이제 막 프로젝트를 시작하는 팀에 합류해 기술 스택부터 하나하나 결정해 나가는 것을 지켜볼 수 있었는데, 시간이 촉박해서 코드 컨벤션은 상세히 정하지 못한 채 프로젝트에 들어가게 되었다. 매주 코드 리뷰를 하며 사람마다 코드 스타일이 이렇게까지 다를 수 있다는 것을 깨달았다. 물론 차근차근 읽으면 이해할 수 있는 내용들이었지만, 소스 코드의 규모가 점차 커질 수록 내 코드는 물론이고 다른 팀원의 코드들도 점차 난해해졌다.

이후 선배들이 팀 내 코드 컨벤션을 정립해 나갈 수록 코드가 깔끔해지고 의사소통이 수월해졌다. 코드 컨벤션을 정립하기 전 코드 리뷰를 통해 단순히 짧고 효율 좋은 코드라해도 남이 모르고 이후에 봤을 때 주석 없이는 알 수 없으면 소용 없다는 것도 배웠다.

이번에 새로운 사이드 프로젝트를 시작하기 전 코드 컨벤션을 정하고 가기로 했다. 유명한 구글의 자바 코드 컨벤션을 따를 생각이다. 영어로 된 문서이고, 은근히 내용도 길어서 내용 공유 겸 아래에 요약한 내용을 정리해보았다.

구글 코드 컨벤션

소스파일

  • 파일 이름은 최상위 클래스의 대소문자 구분 이름 + .java 확장자로 한다.
  • 인코딩은 UTF-8 로 한다.
  • 줄바꿈 문자(\n) 제외 스페이스(\0x20)가 유일한 공백 문자이며 탭(\t) 은 들여쓰기에 사용하지 않는다.
  • 이스케이프 시퀀스가 있는 모든 문자는 유니코드가 아닌 이스케이프 시퀀스를 사용한다. (\n, \b, \t 등)
  • Non-ASCII 문자의 경우 실제 유니코드 문자 또는 유니코드 이스케이프가 사용된다. (μs, \u03bsc 등) 프로그램이 ASCII 문자를 처리하지 못할 수 있다는 생각에 코드 가독성을 낮추지 마라.

소스파일 구조

소스파일은 아래 순서대로 작성한다. 또 각 섹션은 하나의 빈 줄로 구분한다.

  1. (있다면) 라이센스, 저작권 정보
  2. package 구문
  • 아무리 길어도 줄바꿈하지 않는다.
  1. import 구문
  • 아무리 길어도 줄바꿈하지 않는다.
  • 와일드카드(*) 를 사용하지 않는다.
  • static import -> non-static import 순서대로 기재한다.
  • 만약 한 블록({ }) 내에 static 과 non-static import 가 모두 있는 경우 하나의 빈 줄로 두 블록을 구분한다.
  • 정적 중첩 클래스는 static import 로 가져오지 않는다.
  1. 하나의 최상위 클래스
  • 클래스의 멤버와 initializer 의 순서는 따로 없지만, 논리적인 순서를 따라야 한다. 새 메서드를 끝에 추가하는 것은 시간순이지 논리적인 순서가 아니다.
  • 그러나 같은 동일한 이름의 메서드는 연속적으로 작성한다. 여러 생성자도 마찬가지다.

포맷

중괄호

  • 구현부가 없거나 한 줄의 구문을 포함해도 중괄호를 사용한다.
  • 중괄호는 K&R 스타일 을 따른다.
  • 여는 중괄호 뒤와 닫는 중괄호 앞쪽을 줄바꿈한다. 닫는 중괄호 뒤쪽은 구문이 끝나거나 메소드/생성자/이름 있는 클래스의 내용이 끝났을 때만 줄바꿈한다.
  • 빈 블록은 줄바꿈하지 않고 {} 로 사용할 수 있다. 그러나 multi-block(if~else, try~catch~finally) 은 할 수 없다.
return () -> {
    while(condition()) {
        method();   //한 줄이어도 중괄호를 사용한다.
    }
}

return () -> {} //빈 블록은 줄바꿈하지 않아도 된다.

return new MyClass() {
    @Override public void method() {
        if(condition()) {   //여는 중괄호 뒤쪽을 줄바꿈함
            try {
                somthing();
            } catch (Exception e) {}    //잘못된 케이스. 멀티 블록은 {} 로 사용할 수 없다.
        } else if (otherCondition()) {  //닫는 줄바꿈 뒤쪽은 구문이 끝나지 않아 줄바꿈하지 않음.
            somethingElse();
        } else {
            lastThing();
    }
};  //세미콜론은 줄바꿈하지 않는다.

들여쓰기와 줄바꿈

  • 들여쓰기는 2칸씩 늘어난다.
  • 한 줄에는 하나의 명령문만 사용하고 다른 명령문은 줄바꿈한다.
  • 한 줄은 100자가 넘어가면 줄바꿈한다. 단, 이 제한을 준수할 수 없거나(JavaDoc의 긴 URL, 긴 JSNI 메소드 등) package, import 문, 주석으로 작성한 쉘 스크립트에 복사할 수 있는 커맨드 라인은 예외다.
  • 코드의 길이가 페이지를 넘어갈 때 줄바꿈 하는 것을 (Line-wrapping) 이라고 한다.

회고

회사에서 프로젝트는 혼자 하는 것이 아니다. 내 코드를 항상 남들이 보고, 남의 코드를 항상 내가 본다. 일관된 코드 스타일을 따르지 않으면 스파게티 코드가 되고, 그렇다고 나만의 스타일을 고수하면 다른 사람들이 내 코드를 이해할 수 없게 된다.

다른 팀원과의 의사소통과 배려를 위해서라도, 시간이 조금 걸리지만 코드 컨벤션을 준수하며 코딩하는 습관을 길러야겠다.
